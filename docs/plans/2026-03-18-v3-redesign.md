# 2026-03-18 v3 重新设计方案

## 背景
v2 存在采集与推送耦合、无智能过滤、无可观测性三大问题，导致 OpenClaw cron 超时频繁。

## 核心决策记录

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 用户范围 | 单用户先行，user_id 预留多用户 | 当前需求明确，但表结构不锁死 |
| 数据源扩展 | 配置驱动，t_data_sources 注册 | 新增源无需改代码 |
| 路由模式 | 多播（Multi-cast） | 一条内容可命中多个频道 |
| 输出格式 | 当前统一格式，output_format 字段预留 | 先跑通，再做频道专属输出 |
| 表命名 | t_ 前缀，所有字段含 COMMENT | 规范统一，降低维护成本 |
| 任务粒度 | 主任务(task_id) + 子任务(sub_task_id) | 主任务=完整链路，子任务=阶段×数据源/频道 |
| LLM 参与范围 | 仅分析层，推送链路零 LLM | 推送稳定性 > 推送质量，LLM 不参与关键路径 |
| 失败原因分类 | error_type 枚举（6类） | 便于统计哪类问题最多，指导优化 |

## 涉及变更

- 新增：`filter/` 分析层模块
- 新增：`router/` 路由层模块  
- 新增：`push/run_push.py` 独立推送入口
- 新增：`config/channels.yaml` + `sources.yaml`
- 重写：`db/schema.sql`（t_ 前缀，8张新表）
- 迁移：旧 `github_trending/hackernews/ai_news` → `t_raw_items`
- 修改：OpenClaw cron payload 去除采集步骤

## 实现优先级

1. Phase 1（紧急）：push/run_push.py + OpenClaw cron 修复，解决当前超时
2. Phase 2：Schema 迁移
3. Phase 3：filter 分析层
4. Phase 4：router 路由层
5. Phase 5：全链路可观测性
